# Alembic — new-stack Postgres schema migrations

Versioned schema migrations for the **new stack's** PostgreSQL (the system of
record after cutover). `target_metadata` is `backend/models.py`'s `Base.metadata`
(the 64 real tables). The SME/derived **VIEWs** are NOT modelled here — they live
in `models.SME_AND_DERIVED_VIEWS` and are created by `dual_ci`, so autogenerate
ignores them.

Run from the repo root; the URL comes from `$DATABASE_URL` (env.py normalises it
to psycopg2 — Alembic is synchronous):

```bash
export DATABASE_URL=postgresql+psycopg2://<user>:<pass>@<host>:5432/gihub
alembic -c backend/alembic.ini current            # where am I?
alembic -c backend/alembic.ini history            # migration list
```

## How Alembic and dual_ci divide the work
- **`dual_ci`** owns the *initial* load: it creates the schema from `models.py`
  (+ the views) and copies the SQLite data. Used once, at cutover (and it WIPES
  the target — see `docs/DEPLOY.md` §4).
- **Alembic** owns *incremental* changes after that. Because the baseline was
  autogenerated from the same `models.py`, `dual_ci`'s schema matches it exactly
  (`alembic check` → "No new upgrade operations detected").

## At cutover (once): stamp, don't upgrade
`dual_ci` already created the tables, so mark the baseline as applied rather than
re-running it:
```bash
alembic -c backend/alembic.ini stamp head
```

## On a fresh, empty DB instead: upgrade
```bash
alembic -c backend/alembic.ini upgrade head        # creates the 64 tables
# (then create the SME/derived views separately — they're not Alembic-managed)
```

## Making a schema change afterwards
1. Edit `backend/models.py` (add a column/table/index, etc.).
2. `alembic -c backend/alembic.ini revision --autogenerate -m "what changed"`
3. **Review** the generated file in `versions/` (autogenerate is a draft — check
   it, especially data-affecting changes).
4. `alembic -c backend/alembic.ini upgrade head`
5. Keep `models.py` and the migration in the same commit.

## Guardrails
- Never point Alembic at the live **SQLite** DB — this is Postgres-only.
- `alembic check` in CI/pre-deploy catches "models changed but no migration".
- The `alembic_version` table is Alembic's bookkeeping; `dual_ci`'s 64-table
  parity ignores it.
