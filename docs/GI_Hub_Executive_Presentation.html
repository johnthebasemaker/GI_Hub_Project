Why this exists

The problem we set out to solve

  • Disconnected tools. Stock, procurement and the lining estimator each lived in separate spreadsheets — numbers never agreed.
  • No accountability. Who issued what, to whom, and who approved it was lost in email and WhatsApp.
  • Stale planning. The estimator showed quantities that didn't reflect what was actually consumed or received.
  • Manual, error-prone. Re-keying SAP codes and quantities invited mistakes that surfaced only at audit.

Accountability in action

Daily issue → End-of-Day commit

The control point of the whole system. Nothing touches the permanent ledger until a HOD signs off.

  • Store Keeper logs issues during the day — each line auto-stamped with who, what, when.
  • Lines sit in a pending state — reviewable, editable, rejectable.
  • HOD reviews, approves or rejects per-line, then commits End-of-Day.
  • On commit, stock is debited and the line becomes an immutable, audited consumption record.
  • Rejected lines are archived (not deleted) — the audit trail is never broken.

The headline integration

Smart Material Estimator, merged in

A standalone engineering tool for Rubber & Brick lining projects is now a first-class portal inside the ERP — locked to HOD & Admin. It answers: "For this scope of work, what materials do we need, what do we have, and what must we buy?"

🏗️ Equipment

Tanks, vessels & their lining systems with surface areas (SQM) per site.

🧮 Recipes

Material consumption per 1 SQM for every lining system code.

📊 Coverage

Demand vs. availability, coverage % and shortfall — per equipment, location & system.

🛒 Procurement

Smart suggestions and a "days of continuation" runway from current stock.

Apple-to-apple parity: the estimator runs the original tool's own code as a literal drop-in — every chart, report and layout preserved — bridged to the ERP database through compatibility views. Same experience, one source of truth.
The clever part

Strict isolation, live numbers

The estimator's inventory is kept separate from the ERP stock ledger to prevent clutter and corruption — yet its quantities stay current by rolling up real ERP movements at read time.

Available = Initial  +  Received  −  Consumed
  • Initial — the SME-owned baseline, loaded once from the engineering Excel files.
  • Received / Consumed — pulled live from the ERP ledger, never re-keyed.
  • Computed on the fly by a SQL view — nothing is stored, so it can never drift.
  • SME write operations never touch the ERP inventory table, and vice-versa.

Newest module

Man-Hours & Labor Tracking

The newest addition tracks labor the way the estimator tracks material: who worked, where, and for how long — then compares planned vs. actual man-hours to expose over-runs. Locked to HOD & Admin, fully site-scoped, and isolated from the stock ledger.

👥Roster

OWN (GI) staff & Supply subcontractors per site, with designation & company.

🕒Timesheets

One-click Excel upload or a per-day grid; In/Out auto-computes Normal + Overtime hours.

📐Estimator

Define the required man-hours per equipment / system — the labor budget for a scope.

📊Variance

Estimated vs. actual, over-runs flagged red, with a reason-capture field per scope.

🧮SQM productivity

Record the team's completed SQM and distribute it across the crew — evenly or by hours worked — for true labor productivity.

🧑‍🔧Employee-wise view

A clean, date-by-date timeline of exactly where each worker spent their hours — no clutter.

Same DNA as the material estimator: it reads the equipment & system catalogue but writes only its own isolated tables — never the stock ledger or End-of-Day path. Proven on real data: 22 employees · 209 timesheets imported and verified in one pass.
Trust the numbers

Built for integrity

🧪Automated regression harness

554 checks cover schema, every workflow state machine, RBAC locks and the SME integration. Runs before every release; the build is gated on green.

🧾Immutable audit trail

Posted ledger rows are never edited in place. Corrections happen via reversal documents; rejected items are archived, not deleted.

📐Derived, not declared

Live stock and estimator availability are computed from movements — eliminating the "someone typed the wrong number" class of error.

🔒Least-privilege RBAC

Page access is exact-locked per role; multi-site scoping prevents cross-contamination of data between HQ and CNCEC.

Self-healing schema: the database migrates itself forward on startup — new columns, views and seed data apply idempotently with zero manual DB surgery.

Going to production

Production-grade deployment

The platform is being packaged for a dedicated company server — fully containerized, reverse-proxied with TLS, continuously deployed, and wired to WhatsApp Business — while the local development setup keeps working untouched.

🐳Docker

The whole stack — app, AI, proxy, backups — comes up with one docker compose up. Reproducible on any Linux server.

☁️Hetzner + Nginx

A cost-efficient cloud server behind an Nginx reverse proxy with TLS. AI models stay on a private network — never exposed to the internet.

⚙️CI/CD

Push to GitHub → the server auto-pulls, rebuilds & restarts. Zero manual deploys, every change versioned.

💬WhatsApp Cloud API

Direct Meta Business messaging for alerts & two-way replies, with a secure webhook receiver.

🧠Private AI sidecar

FastAPI + a local LLM serve the manual assistant on-box — company data never leaves the server.

💾Automated backups

Nightly snapshots with off-box retention and a documented one-command restore path.

Memory-efficient by design: tuned to run multiple AI models on modest hardware without contention — one model resident at a time, the rest of the stack kept lean.
