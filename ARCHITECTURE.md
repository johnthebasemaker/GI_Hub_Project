# ⚡ General Industries Lightning Hub v2.0 (CNCEC-System)
**Enterprise Inventory, Logistics & Tracking System**

## 📖 Project Overview
This is a multi-site enterprise resource planning (ERP) application built for heavy industry material tracking. It facilitates real-time inventory management, cross-site material requests, role-based approval workflows, and automated background notifications.

**Core Tech Stack:**
* **Frontend:** Streamlit (Custom CSS injected for Dark/Navy/Gold branding)
* **Backend:** Python 3.12
* **Database:** SQLite3 (Local DB file)
* **Data Processing:** Pandas, OpenPyXL
* **Automation:** PyWhatKit (WhatsApp), smtplib (Email)
* **Reporting:** fpdf2 (PDF generation)
* **Authentication:** bcrypt (Hash-based passwords)

---

## 📂 File Architecture
The project is modularized to separate UI, database transactions, background workers, and reporting.

* `main.py`: The entry point. Handles routing, session state checks, and renders the main dashboards (Admin Portal, HOD Portal, Live Inventory, Daily Issue Log).
* `database.py`: The single source of truth for all SQL transactions. Handles connection pooling, table initialization, EOD commits, and the WhatsApp message queue.
* `auth.py`: Handles all RBAC (Role-Based Access Control), bcrypt password hashing, login forms, and user registration requests.
* `ui_components.py`: Contains custom CSS injections (hiding default Streamlit headers, styling the sidebar toggle) and rendering functions for KPIs and Plotly charts.
* `mailer.py`: Handles SMTP automated emails for logistics and EOD report delivery. Includes OS-aware checks (`win32com` for Windows, `mailto:` for Mac).
* `reports.py`: Uses `fpdf2` to generate immutable, branded PDF records (e.g., Purchase Requests, Universal Table Exports).
* `whatsapp_worker.py`: An asynchronous background script that constantly scans the SQLite `whatsapp_queue` table and executes local browser automation to send notifications without blocking the Streamlit UI.
* `config.py`: Stores global variables, brand colour hex codes, and role hierarchy definitions.

## 🔐 Role-Based Access Control (RBAC)
The system operates on strict data isolation and role hierarchy.
1. **Admin (HQ):** Global "God-View". Can manage all users, edit the Master Database, view global inventory, approve cross-site transfers, and view audit logs.
2. **Head of Department (HOD):** Site-specific management. Can view their specific site's inventory, commit staging queues to the master log, upload PRs, and request materials from other sites.
3. **Supervisor:** Can view live stock and staging queues.
4. **Floor Worker:** Restricted to the "Daily Issue Log". Scans materials and adds consumption data to the staging queue for HOD review.

## ⚙️ Core Workflows & Logic

### 1. The Consumption Loop (Daily Issue Log)
* Workers scan/enter materials used into `pending_issues` (Staging Queue).
* HODs review the staging queue in their portal and click "Commit".
* `commit_eod` moves records from `pending_issues` to `consumption` and deducts from `inventory.Current_Stock`.

### 2. Cross-Site Material Requests
* HOD requests material from Site B.
* Request lands in `cross_site_requests` (Status: pending).
* Admin reviews and approves.
* **Automation:** Admin notes are saved, and the `queue_whatsapp_alert` function writes to the DB, triggering `whatsapp_worker.py` to notify the requesting HOD.

### 3. Purchase Requests (PR) & Logistics
* HOD uploads a PR PDF or logs PR details into `pr_master`.
* When physical goods arrive, Site receives against the PR into the `receipts` table.
* The system calculates `Requested_Qty - Received_Qty = Pending_Qty`.
* **Automation:** An email is dispatched to Logistics alerting them of partial or full delivery.

### 4. Enterprise Reporting & Audit
* **Universal PDF:** Any database table can be exported as a branded PDF (`reports.py`).
* **Audit Ledger:** Every critical action (Approval, Rejection, DB Edit) is stamped in `system_audit_log` with User ID, Timestamp, and Action Type.

## 🗄️ Database Schema Summary
SQLite file automatically generated by `init_db()` in `database.py`.

* `users`: Active system accounts.
* `pending_users`: Registration queue.
* `inventory`: Master material list (SAP Code, UOM, Limits).
* `consumption`: Permanent record of all issued materials.
* `pending_issues`: Temporary staging queue before EOD commit.
* `pr_master`: Open purchase requests.
* `receipts`: Inbound deliveries.
* `cross_site_requests`: Material transfer workflows.
* `whatsapp_queue`: Pending and sent automated messages.
* `system_audit_log`: Immutable enterprise event tracking.
